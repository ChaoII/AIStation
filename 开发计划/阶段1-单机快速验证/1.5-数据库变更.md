# 阶段1：单机快速验证 — 数据库变更

## 无需新增表

阶段1 完全复用现有数据表：

| 表 | 用途 | 状态 |
|----|------|------|
| `video_algorithms` | 算法定义（模型路径/运行时配置/参数预设） | ✅ 已有 |
| `video_algorithm_tasks` | 算法任务（绑定摄像头+算法+调度配置） | ✅ 已有 |
| `video_alarm_rules` | 告警规则（触发条件/通知方式） | ✅ 已有 |
| `video_alarm_records` | 告警记录（含 ai_result 字段存检测数据） | ✅ 已有 |

## 字段说明

### AlgorithmTaskModel.status

推理调度器使用 `status` 字段：

| 值 | 含义 | 调度器行为 |
|----|------|-----------|
| RUNNING | 应运行推理 | 检查 Worker 进程，未启动则启动 |
| STOPPED | 应停止推理 | 检查 Worker 进程，运行中则停止 |
| ERROR | 异常（Worker 崩溃） | 保留人工处理，不自动重启 |

### AlarmRecordModel.ai_result

回调处理时写入完整的检测数据：

```json
{
  "task_id": 1,
  "algorithm_type": "INTRUSION",
  "detections": [
    {"label": "person", "confidence": 0.92, "bbox": {"x":0.1, "y":0.2, "w":0.3, "h":0.5}}
  ],
  "frame_timestamp": "2026-05-31T10:00:05.789Z",
  "inference_latency_ms": 12.3
}
```

### AlarmRuleModel.algorithm_task_id

已有字段，用于将告警规则关联到特定算法任务。Callback 处理时会：
1. 根据 camera_id + alarm_type 查找匹配的规则
2. 创建告警记录时记录 `rule_id`

## 数据迁移注意事项

- 已有数据的 `video_algorithms.runtime_config` 可能为 NULL，启动 Worker 时需提供默认值
- 默认 runtime_config：`{"engine": "onnxruntime", "gpu": {"enabled": false}}`
- 默认 preset_params：`{"conf_threshold": 0.5, "nms_threshold": 0.45}`
- 如果 `AlgorithmTaskModel.schedule_json` 为空或无 `slots`，视为全天候运行
