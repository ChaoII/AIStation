# 阶段3：规模化运营 — 高可用设计

## 1. Web 主站高可用

### 1.1 无状态设计

FastapiAdmin Web 主站**所有运行时状态存储在 Redis/PostgreSQL 中**，自身不保留本地状态：

| 状态 | 存储位置 | 说明 |
|------|----------|------|
| 用户 Session | Redis | JWT 无状态，无需 session |
| 限流计数器 | Redis | FastAPILimiter |
| 推理进程表 | 不存 → 分布式下无 `_running_inferences` | 改为 DB 中的 AiBoxTaskModel |
| WebSocket 连接 | 内存（每个实例独立） | WS 断线由前端自动重连 |

### 1.2 水平扩展

### 1.3 WebSocket 粘性

使用 Nginx `ip_hash` 确保同一客户端的 WS 连接落在同一实例：

```nginx
upstream websocket {
    ip_hash;
    server 10.0.1.1:8001;
    server 10.0.1.2:8001;
}
```

或使用 Redis Pub/Sub 跨实例广播：

```python
# 跨 Web 实例广播告警
class AlarmWSManager:
    async def broadcast(self, data: dict):
        # 1. Redis publish
        await redis.publish("alarm:channel", json.dumps(data))
        # 2. 本地广播
        await self._local_broadcast(data)

# 每个 Web 实例订阅 Redis channel
async def subscribe_alarm_channel():
    pubsub = redis.pubsub()
    await pubsub.subscribe("alarm:channel")
    async for message in pubsub.listen():
        if message["type"] == "message":
            data = json.loads(message["data"])
            await alarm_manager._local_broadcast(data)
```

## 4. AI 盒子故障转移

### 4.1 故障检测

| 检测方式 | 延迟 | 说明 |
|----------|------|------|
| 心跳超时 | ~3min | 连续 3 次心跳未收到 |
| Web 主动探测 | ~10s | 定期 GET /health（可选） |
| Worker 回调超时 | ~30s | 回调连续失败标记盒子可疑 |

### 4.2 故障转移流程

```
盒子心跳超时 (180s)
  │
  ▼
Web 标记盒子 OFFLINE
  │
  ▼
调度器触发自动迁移：
  for each RUNNING task on OFFLINE box:
    1. 选择一个 ONLINE 盒子（按负载均衡策略）
    2. 更新 AiBoxTaskModel.box_id = new_box_id
    3. 更新 local_status = PENDING
    4. 新盒子下次心跳收到 start_task 命令
  │
  ▼
Agent 重新上线时：
  1. 不再持有已迁移的任务（本地 tasks/ 目录中已删除）
  2. 如有残留 Worker，由 Agent 清理
```

### 4.3 手动故障转移

```python
# 运维 API
POST /api/v1/algorithm/box/{id}/drain
# 1. 标记盒子 DRAINING（不再分配新任务）
# 2. 逐步迁移现有任务
# 3. 所有任务迁移完成后，标记 OFFLINE
```

## 5. 数据一致性

| 场景 | 保证方式 |
|------|----------|
| 任务重复启动 | Agent 幂等：配置已存在则 skip |
| 任务丢失 | Agent 心跳上报全量任务列表，Web 核对下发补充 |
| 回调重复 | AlarmService.interval_seconds 去重 |
| 盒子重复注册 | box_id 唯一约束，新 token 覆盖旧 token |
| 任务配置版本不一致 | Agent 上报 config hash，Web 比对后重发 |

## 6. 灾难恢复

### 6.1 备份策略

| 数据 | 备份频率 | 保留 | 方式 |
|------|----------|------|------|
| PostgreSQL | 每 6h | 30天 | pg_dump / WAL 归档 |
| 模型文件 | 每次上传 | 保留所有版本 | 文件系统 + 对象存储 |
| 检测快照 | - | 7天 | 定期清理 + 可选 S3 归档 |
| 告警记录 | - | 90天 | 超过 90 天自动归档到冷存储 |

### 6.2 恢复步骤

```bash
# 1. 恢复 PostgreSQL
pg_restore -d video_surveillance latest_backup.dump

# 2. 恢复模型文件
rsync -av backup/models/ data/models/

# 3. 重启服务
systemctl restart fastapiadmin
# AI 盒子 Agent 自动重新注册
```
